# finalize-scenario — конвертация черновика в @playwright/test

Переводит `draft.js` (черновик фазы 1) в детерминированный чистовик `final.spec.ts`,
который гоняется быстро и без агента.

## Аргумент
Имя сценария. Например: `/finalize-scenario create-order`.
Если не передан — покажи папки в `scenarios/`, у которых есть `draft.js`, и спроси какую финализировать.

## Предусловия
- Есть `scenarios/<имя>/draft.js`. Если нет — скажи, что сначала нужен `/test-scenario <имя>`.
- Прочитай `scenarios/<имя>/scenario.md` (для имени теста и `baseUrl`) и последний отчёт
  в `reports/` (чтобы понимать, какие шаги подтверждены).

### СТОП-условие: фаза 1 должна быть пройдена ПОЛНОСТЬЮ

Посмотри в последнем отчёте строку **Итог**. Собирать чистовик можно только при
`✅ пройден полностью`. Если там `⚠️ частично (N из M)` или `❌ провал на шаге K` —
**не финализируй**, покажи пользователю, какие шаги отвалились (в отчёте есть колонка
«пробовал: …»), и предложи разобраться или перепройти `/test-scenario <имя>`.

Почему это жёстко: у фазы 1 правило «в `draft.js` попадают только подтверждённые шаги».
Значит провалившийся шаг не ломает черновик — он из него **молча исчезает**. Соберёшь чистовик
из такого черновика — получишь ЗЕЛЁНЫЙ тест, который просто больше не проверяет сломанное.
Пользователь сгенерированные файлы не читает, поэтому увидит `1 passed` и решит, что всё хорошо.
Частичный проход — это не «почти получилось», это «шаги пропали, нужен человек».

## Что делать

1. **Извлеки блоки.** Возьми из `draft.js`:
   - `ensureAuth` — из блока между `// >>> AUTH` и `// <<< AUTH`;
   - шаги — из блока между `// >>> STEPS` и `// <<< STEPS`.
   Боилерплейт (`chromium.launch`, `newPage`, `try/finally`, `loadCreds`) в чистовик как есть
   НЕ переносится — его роль берёт на себя тест-раннер; `loadCreds` переезжает в хелпер (см. п.2).

2. **Вынеси авторизацию в хелпер `scenarios/_auth.ts`** (один на все сценарии), туда же `loadCreds`
   (креды по-прежнему из `scenarios/auth.json`, с приоритетом env
   `TEST_USER`/`TEST_PASS`). Сигнатура: `export async function ensureAuth(page: Page): Promise<void>`.

2а. **`test` и `expect` импортируй из `../_fixtures`, а НЕ из `@playwright/test`.**
   Там автофикстура, которая молча собирает сетевые (HTTP ≥ 400, упавшие запросы) и консольные
   (`pageerror`, `console.error`) ошибки, прикладывает их к отчёту и печатает одну строку-сводку.
   Тест она не роняет — это диагностика, а не проверка.

   Зачем: Playwright сам про ошибки API ничего не знает, он проверяет только то, что написано
   в `expect()`. Сломанный бэкенд поэтому проявляется как «элемент не найден» или таймаут —
   вывод показывает пальцем на локатор, хотя причина в 500. Фикстура закрывает этот разрыв.
   Ронять тест из-за сетевых ошибок НЕ надо: чужие 404 и шум аналитики сделали бы прогон
   красным на ровном месте, и ему перестали бы верить.

   `_fixtures.ts`, как и `_auth.ts`, — файл, который ведут руками; он переживает пересборку.

3. **Собери `final.spec.ts`** по шаблону (авторизация — в `beforeEach`, т.к. storageState не используем,
   проверяем каждый раз).

   **В начале файла — обязательный комментарий-шапка с командой запуска.** Чистовик читают и
   гоняют без агента и без этого скилла, поэтому способ его запустить должен лежать прямо в файле,
   а не в README или в отчёте. Минимум — запуск именно этого сценария; при наличии добавь то,
   что нужно знать перед прогоном (нестандартный таймаут, побочные эффекты, переменные окружения):
   ```ts
   /**
    * Запуск:
    *   npx playwright test scenarios/<имя>/final.spec.ts           # только этот сценарий (headless)
    *   npx playwright test                                          # все сценарии
    *   npx playwright test scenarios/<имя>/final.spec.ts --headed   # с ВИДИМЫМ браузером
    *   npx playwright test scenarios/<имя>/final.spec.ts --debug    # видимый браузер + пошаговый Inspector
    *   npx playwright test scenarios/<имя>/final.spec.ts --ui       # UI Mode: разбор ЗАПИСИ прогона, не живой браузер
    *
    * Переопределить стенд: BASE_URL=http://… npx playwright test …
    * Креды: scenarios/auth.json (или env TEST_USER / TEST_PASS).
    */
   ```
   Про `--headed` / `--ui` в шапке пиши именно так, с пояснением: по умолчанию раннер работает
   headless, и это первое, обо что спотыкаются («запустил — браузера нет»). А `--ui` показывает
   не живой браузер, а запись прогона — без пометки его принимают за режим наблюдения.
   ```ts
   import { Locator } from '@playwright/test';       // типы — отсюда
   import { test, expect } from '../_fixtures';      // НЕ из '@playwright/test'
   import { ensureAuth } from '../_auth';

   test.beforeEach(async ({ page }) => {
     await ensureAuth(page);          // идемпотентный логин перед каждым тестом
   });

   test('<name из scenario.md>', async ({ page }) => {
     // <сюда — сконвертированные шаги; baseURL берётся из playwright.config.ts>

     // Последней строкой теста — ссылка на экран, где прогон закончился:
     console.log(`Последний экран: ${page.url()}`);
   });
   ```

   **В конце теста печатай ссылку на последний экран.** Раннер закрывает браузер сразу после
   теста (`page`/`context` — фикстуры, их сносит тирдаун), и посмотреть глазами на результат
   не успеваешь даже с `--headed`. Одна строка `console.log` с `page.url()` решает это: ссылку
   видно в консоли и можно открыть руками. Если по ходу сценария есть другие важные точки
   (созданная сущность, промежуточный результат) — печатай и их.

   **Чего для этого делать НЕ надо:** не оставляй браузер открытым через `page.pause()`.
   Пауза действительно держит окно, но безусловный вызов вешает обычный `npx playwright test`
   навсегда в ожидании Resume — тест перестаёт быть автоматическим. Прятать её за флагом вроде
   `KEEP_OPEN` тоже не нужно: ссылки в консоли достаточно, а `--debug` покрывает случай, когда
   действительно надо походить по шагам живьём.

4. **Сконвертируй проверки** черновика в идиоматичные `expect()` (это и есть «перевод в
   команды обычного инструмента»). Таблица маппинга:

   | Черновик (фаза 1, бросает при провале) | Чистовик (@playwright/test) |
   |---|---|
   | `await page.waitForURL(/re/)` | `await expect(page).toHaveURL(/re/)` |
   | `await loc.waitFor({ state: 'visible' })` | `await expect(loc).toBeVisible()` |
   | `await page.getByText('X').first().waitFor()` | `await expect(page.getByText('X').first()).toBeVisible()` |
   | `if ((await loc.inputValue()) !== 'X') throw…` | `await expect(loc).toHaveValue('X')` |
   | `if ((await loc.count()) < 1) throw…` | `await expect(loc).toHaveCount(N)` / `not.toHaveCount(0)` |
   | `if (await loc.isDisabled()) throw…` | `await expect(loc).toBeEnabled()` |
   | `if (!(await loc.isChecked())) throw…` | `await expect(loc).toBeChecked()` |
   | `page.goto(BASE_URL + '/path')` / `page.goto(BASE_URL)` | `page.goto('/path')` / `page.goto('/')` (baseURL из конфига) |

   **Опрос с паузами переписывай на нативные ожидания.** В черновике фазы 1 нередко встречается
   поллинг вида `for (let i = 0; i < 50 && (await loc.isDisabled()); i++) await page.waitForTimeout(200)`
   или `page.waitForFunction(() => document.querySelector(...)...)` — агент так обходит асинхронный
   UI. В чистовике это заменяется на `expect()`, который ждёт сам:

   | Черновик | Чистовик |
   |---|---|
   | цикл с `waitForTimeout` по `isDisabled()` | `await expect(loc).toBeEnabled({ timeout: … })` |
   | `waitForFunction` по смене текста элемента | `await expect(loc).not.toHaveText(prev, { timeout: … })` |
   | `waitForTimeout(N)` «чтобы подгрузилось» | `expect()` на то, чего реально ждём |

   Голый `waitForTimeout` в чистовике оставлять нельзя: он либо тормозит тест, либо флачит.
   Управляющие конструкции (циклы по `count()`, выход по состоянию UI) — переносятся КАК ЕСТЬ,
   это логика, а не ожидание.

   **Таймаут теста.** Дефолт @playwright/test — 30с на тест. Прикинь по черновику, сколько шагов
   и повторений реально выполняется: длинные сценарии (списки, чек-листы, пагинация) в дефолт не
   влезают. Ставь `test.setTimeout(…)` первой строкой теста с запасом на стенд с бо́льшим объёмом
   данных — прогон от этого не замедляется, а падение по таймауту на ровном месте исключается.
   В шапке-комментарии поясни, почему таймаут такой.

   Действия (`click`, `fill`, `getByText`, `locator(css)`) переносятся как есть — они одинаковы
   в library API и в @playwright/test. Локаторы не переписывай на getByRole — оставляй text+CSS.

5. **Создай/обнови `playwright.config.ts`** (клади рядом со сценариями — в корне `scenarios/`
   или в корне проекта; выбери одно место и держись его). Минимум:
   ```ts
   import { defineConfig } from '@playwright/test';
   export default defineConfig({
     testDir: './scenarios',
     testMatch: '**/final.spec.ts',
     use: {
       baseURL: process.env.BASE_URL || '<baseUrl из scenario.md>',
       trace: 'on-first-retry',
       screenshot: 'only-on-failure',
     },
     reporter: [['list'], ['html', { outputFolder: 'scenarios/_playwright-report', open: 'never' }]],
     retries: 1,
   });
   ```
   `testMatch: '**/final.spec.ts'` гарантирует, что раннер подхватывает только чистовики и
   игнорирует `draft.js`.

6. **Прогони чистовик для проверки:** `npx playwright test scenarios/<имя>/final.spec.ts`.
   - Прошёл → отлично. Запиши краткий отчёт о финализации в `scenarios/<имя>/reports/<дата_время>-final.md`
     (имя = дата_время через `date +"%Y-%m-%d_%H-%M-%S"`).
   - Упал → покажи вывод раннера. Частая причина — приложение не поднято по `baseURL`, либо
     локатор оказался хрупким. Не «чини» проект — либо поправь локатор в чистовике (text+CSS),
     либо предложи перепройти фазу 1 (`/test-scenario <имя>`).

7. **Отчитайся в чат:** путь к `final.spec.ts` и `playwright.config.ts` (markdown-ссылками),
   результат прогона, команда для будущих быстрых прогонов:
   `npx playwright test scenarios/<имя>/final.spec.ts`.

## Важно
- Исходники тестируемого проекта не трогаем.
- Один сценарий = один `final.spec.ts` внутри своей папки. Общий `playwright.config.ts` — один на все сценарии.
