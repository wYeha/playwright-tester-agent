## 1. Ключевая модель: что живёт долго, а что расходник

**Самое важное в документе.** Если запомнить один раздел — пусть будет этот.

| Файл | Кто ведёт | Судьба |
|---|---|---|
| `scenarios/auth.json` | **человек** | креды, свои у каждого проекта, в `.gitignore` |
| `scenarios/<имя>/scenario.md` | **человек** | живёт долго, правится руками |
| `{.claude,.cursor}/agents/playwright-tester.md` | кит | обновляется через `update` |
| `{.claude,.cursor}/skills/*/SKILL.md` | кит | обновляется через `update` |
| `scenarios/_fixtures.ts` | кит | обновляется через `update` |
| `scenarios/_auth.ts` | фаза 2 создаёт, дальше человек | живёт долго |
| `playwright.config.ts` | леса, дальше человек | `update` не трогает |
| `scenarios/<имя>/draft.js` | агент | **перезаписывается** фазой 1 |
| `scenarios/<имя>/final.spec.ts` | агент | **перезаписывается** фазой 2 |
| `scenarios/<имя>/reports/*.md` | агент | копятся, история |

Запись `{.claude,.cursor}/…` означает **два файла-зеркала**: один читает Claude Code, другой —
Cursor. Тела совпадают, различается фронтматтер. Правите одно — синхронизируйте другое.

Из модели следует правило, которое экономит нервы:

> **Правите `final.spec.ts` или `draft.js` руками — правка испарится при следующей пересборке.
> Молча. Если правка нужна навсегда, её место в `SKILL.md`, в инструкциях агента или
> в `scenario.md`.**

Почему `scenario.md` переживает рефакторинг, а тест — нет: сценарий написан **намерениями**
(«нажать «Сохранить»»), тест — **локаторами** (`.card__title`). Переименовали класс — тест умер,
намерение живо.

## 2. MCP-сервер: порядок важен

Фаза 1 требует живого Playwright MCP-браузера. Здесь легко потерять полчаса.

### Шаг 1. Поднять сервер в отдельном терминале

```bash
npm run playwright:mcp
```

Терминал не закрывать — сервер работает на переднем плане. Успешный старт:
`Listening on http://localhost:8931`.

Сервер запускается в **headed**-режиме: окно видно, за агентом можно наблюдать. На WSL нужен
работающий WSLg — проверить: `echo $DISPLAY` должно выдать `:0`.

### Шаг 2. Только ПОСЛЕ этого запускать Claude Code / Cursor

MCP подхватывается **при старте сессии**. Сервера не было — инструменты `mcp__playwright__*`
не появятся, и перезапуски внутри сессии не помогут.

### Шаг 3. Проверить — по-разному

**Claude Code:**

```bash
claude mcp list
# playwright: http://localhost:8931/mcp (HTTP) - ✔ Connected
```

- `⏸ Pending approval` — согласиться на подключение проектного MCP-сервера (спрашивают при старте
  сессии; защита от того, чтобы склонированный репозиторий молча поднимал сторонние серверы).
- `Skipped — has a "url" but no "type"` — см. ниже про конфиги.

**Cursor:** Reload Window → **Settings → Tools & MCP** → `playwright` → **Enabled**.
Не подключился — **Output → MCP Logs**. Команды `claude mcp list` в Cursor нет.

### Конфиги различаются — это не дублирование по недосмотру

```jsonc
// .mcp.json — Claude Code. Поле "type" ОБЯЗАТЕЛЬНО
{ "mcpServers": { "playwright": { "type": "http", "url": "http://localhost:8931/mcp" } } }

// .cursor/mcp.json — Cursor. Голый url, "type" не нужен
{ "mcpServers": { "playwright": { "url": "http://localhost:8931/mcp" } } }
```

Причина видна из самого сообщения Claude Code: голый `url` неоднозначен — за ним может быть
HTTP-стриминг, SSE или WebSocket. Claude Code отказывается угадывать, Cursor угадывает.
**Без `"type"` Claude Code молча пропускает сервер**, без единого сообщения об ошибке.

## 3. Как писать сценарий

`scenarios/<имя>/scenario.md`, образец — в `scenarios/_TEMPLATE/`.

```markdown
---
name: Создание заказа              # попадёт в test('...')
baseUrl: http://app.local
description: Короткое описание, что проверяем
---

## Предусловия
- Пользователь залогинен (агент авторизуется сам).

## Шаги
1. Открыть главную страницу.
2. Нажать «Создать заказ».
3. Выбрать первую позицию из списка.
4. Заполнить поле «Комментарий» значением "Тест".
5. Нажать «Сохранить».
6. Убедиться, что заказ появился в списке недавних.

## Ожидаемый результат
- В блоке «Недавние заказы» видна карточка «Тест».
```

**Пишите намерения, а не селекторы.** Средне-низкая детализация — это требование, а не
небрежность: локаторы подберёт агент по живому DOM, и подобранное им надёжнее угаданного вами.

Не надо писать:

- **про логин** — агент входит сам, увидев форму (правило 7);
- **про циклы** («прокликать все вопросы») — агент знает (правило 8);
- **про плейсхолдеры** («первая опция, но не «Выберите…»») — агент знает.

## 4. Запуск тестов: флаги

```bash
npx playwright test scenarios/<имя>/final.spec.ts             # headless (по умолчанию!)
npx playwright test scenarios/<имя>/final.spec.ts --headed    # видимое окно
npx playwright test scenarios/<имя>/final.spec.ts --debug     # видимое окно + пошаговый Inspector
npx playwright test scenarios/<имя>/final.spec.ts --ui        # UI Mode: разбор ЗАПИСИ прогона
npx playwright test scenarios/<имя>/final.spec.ts --retries=0 # без ретраев — вдвое быстрее при отладке
BASE_URL=http://другой-стенд npx playwright test              # переопределить стенд
```

Три вещи, на которых спотыкаются:

- **По умолчанию раннер headless.** «Запустил — браузера нет» — это не поломка.
- **`--ui` показывает не живой браузер, а запись прогона.** Это посмертный разбор упавшего теста
  с таймлайном и DOM-снапшотами, а не режим наблюдения. Для наблюдения — `--headed`, но прогон
  занимает секунды и всё пролетит; чтобы разглядеть — `--debug`.
- **Браузер закрывается сразу после теста** — устройство раннера (`page`/`context` — фикстуры,
  их сносит тирдаун), опции «не закрывать» нет. Поэтому тест печатает ссылку на последний экран:

  ```
  Последний экран: http://app.local/orders/64
  ```

  `page.pause()` для этого использовать **нельзя**: безусловный вызов вешает обычный прогон
  навсегда в ожидании Resume.

## 5. Что происходит при ошибках

### Главное

**Playwright не знает про ваши ошибки API.** Он не следит ни за статусами запросов, ни за консолью
браузера — проверяет только то, что написано в `expect()`. Отсюда все неожиданности.

### Типовые падения

**Стенд не поднят** — быстро и однозначно:

```
Error: page.goto: net::ERR_CONNECTION_REFUSED at http://app.local/
   at _auth.ts:27
```

Падает в `beforeEach`, все тесты красные разом. Тест ни при чём.

**Локатор уехал** — понятно:

```
Error: expect(locator).toBeVisible() failed
Locator: getByText('Мои заказы').first()
Timeout: 5000ms
Error: element(s) not found
> 49 |   await expect(page.getByText('Мои заказы').first()).toBeVisible();
```

**API отдаёт 500** — а вот тут коварно:

```
Test timeout of 120000ms exceeded.
Error: locator.click: Test timeout of 120000ms exceeded.
Call log:
  - waiting for locator('label:text-is("Экспресс-доставка")')
```

Слов «500», «API», «network» в выводе **нет**. Выглядит как проблема локатора, хотя лежит бэкенд.

**Как отличить за пять секунд — откройте скриншот падения:**

- каркас страницы цел, **содержимое пустое** → это **данные**, смотрите сеть;
- элемента нет вообще или страница другая → это **вёрстка**, перегоняйте фазу 1.

**Флак** — `1 flaky`: упал, на ретрае прошёл. Это **не** «всё хорошо», это гонка. Трейс от первой
попытки сохранится.

### Сбор сетевых и консольных ошибок

Разрыв закрывает автофикстура `scenarios/_fixtures.ts`. Она **молча** собирает HTTP ≥ 400,
упавшие запросы, `pageerror` и `console.error`, показывает сводку в консоли (до 5 штук) и
прикладывает полный список со стектрейсами к отчёту:

```
⚠ Замечено проблем (сеть/консоль): 1
    [CONSOLE] TypeError: Cannot read properties of undefined (reading 'title')
    Стектрейсы: npx playwright show-report scenarios/_playwright-report → тест → Attachments
```

**Тест она не роняет** — это диагностика, а не проверка. Так задумано: чужие 404 и шум аналитики
красили бы прогон в красный на ровном месте, и ему перестали бы верить.

Почему сводка в консоли, а не только вложение: **вложение в отчёте на зелёном тесте никто никогда
не откроет.** Отчёт открывают, когда что-то упало.

Подключается сама, если `test`/`expect` импортированы из `../_fixtures`:

```ts
import { Locator } from '@playwright/test';   // типы
import { test, expect } from '../_fixtures';  // НЕ из '@playwright/test'
```

### Ваши действия: развилка на четыре ветки

**Не чините тест, пока не поняли причину.**

| Что видите | Что это | Что делать |
|---|---|---|
| `ERR_CONNECTION_REFUSED`, 502 | стенд лежит | поднять приложение |
| Локатор не найден, фича работает | вёрстку поменяли | `/test-scenario` → `/finalize-scenario` |
| Локатор не найден, фича сломана | **регрессия** | **чинить код.** Тест сработал |
| Тест не дошёл до шага, которого в UI больше нет | поведение изменилось | правите `scenario.md`, потом перегон |

Самая опасная — третья. Соблазн «перегнать фазу 1, пусть агент подберёт новые локаторы» велик,
и он сработает: агент обойдёт сломанное место, шаг выпадет из черновика, тест позеленеет.
Ради этого и стоит стоп-условие из раздела 6.

**Непонятно, какая ветка** — отдайте агенту `test-results/<…>/error-context.md`. Playwright
генерирует его сам при падении, и он буквально написан как промпт для ИИ: «объясни, почему тест
упал, и предложи фикс». Внутри — ошибка, стек и текстовое дерево страницы на момент падения.

## 6. Правила, которым следует агент

**Не трогает исходники проекта.** Никаких `data-testid` — только читает DOM и подбирает локаторы
по факту. (Технически это правило держится на инструкции, а не на песочнице.)

**Локаторы — только текст и CSS.** `getByRole`/aria не использует. Порядок: `getByText`/`getByLabel`
→ `locator('css')` по стабильным классам → `filter({ hasText })`.

**Шаг засчитан только по проверке.** Снапшот плюс программная проверка. «Кликнул и вроде норм» —
не считается.

**Правило 7: увидел форму входа — логинится немедленно, независимо от сценария.** На любом шаге,
в любой момент. Следствие: **сценарии, проверяющие сам вход, фазой 1 не гоняются** — агент
залогинится раньше, чем дойдёт до проверки редиректа. Такие спеки пишутся руками.

**Правило 8: повторение с неизвестным числом итераций — это цикл, а не список кликов.** Если агент
прокликал 184 поля на 18 страницах — это факт сегодняшнего прогона, а не структура теста. В код
идёт перебор через `count()` и выход по состоянию UI. Общий принцип: **в код не попадает ничего,
что агент «увидел», — только то, что он вычислил.** Конкретные id, названия, количества — это
данные, они меняются.

**Повторный прогон = полная пересборка черновика.** Старый читается как подсказка по локаторам,
но результат — с нуля.

**Стоп-условие фазы 2.** `/finalize-scenario` собирает чистовик **только** при вердикте
`✅ пройден полностью`. Причина: у фазы 1 правило «в `draft.js` попадают только подтверждённые
шаги», значит провалившийся шаг **молча исчезает** из черновика. Собрать из такого чистовик —
получить зелёный тест, который больше не проверяет сломанное. Частичный проход — это не
«почти получилось», это «шаги пропали, нужен человек».

## 7. Отчёты и артефакты: что когда появляется

| Артефакт | Где | Когда |
|---|---|---|
| HTML-отчёт | `scenarios/_playwright-report/` | **каждый** прогон, перезаписывается |
| Скриншот падения | `test-results/<тест>/test-failed-1.png` | любое падение |
| `error-context.md` | `test-results/<тест>/` | любое падение |
| Трейс (~1 МБ) | `test-results/<тест>-retry1/trace.zip` | **только на ретрае**, т.е. после падения |
| Вложение с ошибками сети | HTML-отчёт | если фикстура что-то поймала |
| `.md`-отчёты сценария | `scenarios/<имя>/reports/` | только `/test-scenario`, `/finalize-scenario` |

```bash
npx playwright show-report scenarios/_playwright-report
```

Папки артефактов в `.gitignore`. **Трейс живёт до следующего прогона** — поймали интересное
падение, разбирайте сразу или скопируйте папку.

Настройки в `playwright.config.ts`: `trace: 'on-first-retry'`, `screenshot: 'only-on-failure'`,
`retries: 1`. Трейс тяжёлый, поэтому пишется не всегда.

## 8. Грабли

**Побочные эффекты на стенде.** Сценарии, которые создают сущности, создают их **по-настоящему**,
каждый прогон. При `retries: 1` упавший тест создаст ещё одну. Гонять только на тестовом стенде;
предупреждение об этом фаза 2 выносит в шапку спеки.

**Сессия MCP-браузера живёт между прогонами.** Поэтому правило 7 может ни разу не сработать —
форма входа просто не появится. Это нормально: `ensureAuth` всё равно проверяется в node-прогоне
черновика, который стартует с чистого контекста.

**`draft.js` не подхватывается раннером** — в конфиге `testMatch: '**/final.spec.ts'`. Намеренно:
черновик запускается только вручную через `node draft.js`.

**Ошибки в консоли браузера тест не роняют.** Если нужно ловить — это отдельная проверка,
которую надо написать осознанно.

**`.playwright-mcp/` растёт.** MCP-сервер пишет туда снимок страницы на каждое действие агента.
Само не чистится, в `.gitignore` есть.

## 9. Claude Code vs Cursor

**Фаза 2 идентична везде** — `npx playwright test` это обычная CLI-команда, ей всё равно, откуда
её запускают. Хоть из голого терминала, хоть из CI.

**Фаза 1 различается:**

| | Claude Code | Cursor |
|---|---|---|
| Конфиг MCP | `.mcp.json`, поле `type` обязательно | `.cursor/mcp.json`, голый `url` |
| Инструкции агента | `.claude/agents/playwright-tester.md` | `.cursor/agents/playwright-tester.md` |
| Скиллы | `.claude/skills/` | `.cursor/skills/` |
| Вызов субагента | Agent tool, `subagent_type` | делегирование `/playwright-tester` |
| Модель прав агента | белый список `tools:` | флаг `readonly:` |
| Включение MCP | согласиться при старте сессии | Settings → Tools & MCP → Enabled + Reload Window |

Корень различий: это **независимые продукты**. MCP стандартизирован, а точки расширения вокруг
него каждый придумывал сам. Отсюда удвоение файлов.

Если слэш-команда не подхватилась — попросите словами: «прогони сценарий `<имя>` через субагента
playwright-tester», «финализируй сценарий `<имя>`». Работу делает один и тот же субагент по одним
и тем же инструкциям, различается только способ его позвать.

> **Честная оговорка.** Cursor-часть описана по устройству конфигов; автор кита гоняет систему
> в Claude Code. Фаза 2 в Cursor работает гарантированно — это обычный CLI.
