# playwright-tester-agent

E2E-тесты, где локаторы подбирает ИИ-агент, а гоняет их обычный Playwright.

Сценарий пишется человеком **на словах**, средне-низкой детализацией. Дальше субагент проходит
его в живом браузере, подбирает устойчивые text/CSS-локаторы и собирает черновик. Второй скилл
превращает черновик в обычную `@playwright/test`-спеку, которая гоняется за секунды **без всякого ИИ**.

Работает в **Claude Code** и **Cursor**.

## Зачем

E2E-тест дорог не в написании, а в подборе локаторов: лезть в DOM, искать устойчивые классы,
проверять, что ничего не матчится дважды. Это скучно и повторяется после каждого рефакторинга вёрстки.

Идея — разделить на две фазы:

| | Фаза 1 «медленная умная» | Фаза 2 «быстрый чистовик» |
|---|---|---|
| **Кто** | субагент `playwright-tester` | скилл `finalize-scenario` |
| **Что** | проходит сценарий в живом браузере, подбирает локаторы | переводит черновик в `@playwright/test` |
| **Сколько** | 3–8 минут | секунды |
| **Результат** | `draft.js` + отчёт | `final.spec.ts` |
| **Когда** | при создании сценария и когда UI изменился | после каждой фазы 1 |

Агент нужен, чтобы **добыть локаторы**, а не чтобы гонять тесты. Реальный замер на сценарии из
18 экранов и 184 полей: фаза 1 — около 7 минут, готовый тест — **15 секунд**.

## Установка

```bash
# 1. развернуть кит
npx github:wYeha/playwright-tester-agent init \
    --base-url http://app.local \
    --login-url http://sso.local/auth
```

```bash
# 2. зависимости — init их только прописывает, но НЕ ставит
npm install
npm run playwright:install
```
```bash
# 3. креды: скопировать образец и вписать user/pass
cp scenarios/auth.example.json scenarios/auth.json
```

```bash
# 4. проверить, что браузер и стенд живы
npm run playwright:verify        # ждём PLAYWRIGHT_OK, status=200
```

```bash
# 5. MCP-сервер — В ОТДЕЛЬНОМ ТЕРМИНАЛЕ, держать открытым
npm run playwright:mcp           # ждём "Listening on http://localhost:8931"
```

```bash
# 6. перезапустить Claude Code / Cursor — и только теперь писать сценарии
```

**Порядок шагов 5 и 6 важен:** MCP подхватывается при старте сессии. Если сервер поднялся,
когда агент уже работал, инструменты не появятся, и перезапуски *внутри* сессии не помогут.

### Параметры `init`

- `--base-url` — адрес приложения. Обязателен.
- `--login-url` — адрес страницы входа. Абсолютный, если авторизация на отдельном домене,
  или относительный (`/login`, `/auth`).

**`--login-url` опускайте только если вход действительно по `/login`** — это дефолт, и он
подставится молча. Если вход у приложения не там, агент пойдёт не туда, а обнаружится это
не сразу. `init` в таком случае печатает предупреждение — не пролистывайте его.

Сам `init` ставить не нужно — `npx` скачает и запустит. Существующие файлы не перезаписываются
(`--force`, если надо), `package.json` и `.gitignore` **дополняются**, а не переписываются.

### Что появится

```
.claude/agents/playwright-tester.md          ← субагент (фаза 1)
.claude/skills/test-scenario/SKILL.md        ← запуск фазы 1
.claude/skills/finalize-scenario/SKILL.md    ← фаза 2
.cursor/…                                    ← то же для Cursor
.mcp.json  .cursor/mcp.json                  ← конфиги MCP (форматы разные!)

scenarios/_fixtures.ts                       ← сбор ошибок сети и консоли
scenarios/_TEMPLATE/scenario.md              ← образец сценария
scenarios/auth.example.json                  ← образец кредов
playwright.config.ts
scripts/playwright-verify.js

.agent-kit.json                              ← хеши файлов, по ним update видит ваши правки
package.json                                 ← дополнен: скрипты + devDependencies
.gitignore                                   ← дополнен: node_modules, артефакты, auth.json
```

`scenarios/_auth.ts` **не создаётся намеренно** — его напишет фаза 2 под форму входа именно
вашего приложения. Скопированный чужой `_auth.ts` выглядел бы правдоподобно и не работал.

`scenarios/auth.json` тоже не создаётся: пароля кит не знает, файл делаете вы из образца.

## Обновление

```bash
npx github:wYeha/playwright-tester-agent update
```

Освежает **только** то, что эволюционирует в ките: агентов, скиллы и фикстуру. Не трогает
`playwright.config.ts`, шаблон сценария и MCP-конфиги — это леса, проект подстраивает их под себя.
Сценарии и `_auth.ts` не трогает никогда.

**Локально изменённые файлы не перезаписывает.** При `init` хеши пишутся в `.agent-kit.json`;
если файл разошёлся — `update` его пропустит и скажет об этом. `--force` перезапишет.

## Требования

Node 18+ и Claude Code или Cursor. Больше ничего — Playwright ставится шагом 2 установки.

## Креды

`scenarios/auth.json` — **свой у каждого проекта**, создаётся из образца (шаг 3 установки).
Вписать нужно **два поля**:

```json
{
  "loginUrl": "http://sso.local/auth",   ← подставил init из --login-url, проверьте
  "user": "CHANGE_ME",                   ← вписать
  "pass": "CHANGE_ME"                    ← вписать
}
```

Адрес приложения здесь **не указывается** — он берётся из `playwright.config.ts`.
Заполнить нужно **до первого `/test-scenario`**: агент полезет логиниться сразу.

> **Файл с паролем лежит в проекте.** От коммита его удерживает только строка
> `scenarios/auth.json` в `.gitignore` — не добавляйте его через `git add -f`.
> Это осознанный размен: взамен у каждого проекта свои креды и свой `loginUrl`.
> Перебить логин и пароль можно через переменные `TEST_USER` / `TEST_PASS` —
> в CI лучше так, файл там не нужен вовсе.

## Цикл работы

```bash
# 1. написать scenarios/<имя>/scenario.md — намерениями, не селекторами
# 2. /test-scenario <имя>        — прочитать отчёт!
# 3. /finalize-scenario <имя>
# 4. npx playwright test scenarios/<имя>/final.spec.ts
```

Про логин в сценарии писать не надо — агент входит сам. Про циклы («прокликать все элементы»)
тоже не надо — агент знает.

## Что живёт долго, а что расходник

Главное, что стоит понять:

| Файл | Кто ведёт | Судьба |
|---|---|---|
| `scenarios/<имя>/scenario.md` | **человек** | живёт долго |
| `scenarios/auth.json` | **человек** | креды, в `.gitignore`, свои у каждого проекта |
| `.claude/…`, `.cursor/…`, `_fixtures.ts` | **кит** | обновляется через `update` |
| `playwright.config.ts`, `_TEMPLATE/` | леса, дальше человек | `update` не трогает |
| `scenarios/_auth.ts` | фаза 2, потом человек | живёт долго |
| `scenarios/<имя>/draft.js` | агент | **перезаписывается** фазой 1 |
| `scenarios/<имя>/final.spec.ts` | агент | **перезаписывается** фазой 2 |

> Правишь `final.spec.ts` или `draft.js` руками — правка **испарится молча** при следующей
> пересборке. Нужна навсегда — её место в `SKILL.md` или в `scenario.md`.

`scenario.md` переживает рефакторинг, потому что написан намерениями («нажать «Сохранить»»),
а тест — локаторами. Переименовали класс: тест умер, намерение живо.

## Обновление сценария при изменении UI

Ничего автоматического — только когда попросите.

| Что случилось | Что делать |
|---|---|
| Кнопку переименовали, поведение то же | `/test-scenario` → `/finalize-scenario` |
| Изменилась логика, появился шаг | сначала правите `scenario.md`, потом обе фазы |
| Фича реально сломана | **чинить код.** Тест сработал как надо |

Тесты **не** обновляются автоматически вслед за кодом. Тест, который сам подстраивается под
изменённый код, перестаёт быть тестом: он всегда зелёный и регрессию не поймает.
